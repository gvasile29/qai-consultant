# A Systematic Literature Review on the Impact of AI Models on the Security of Code Generation

## Abstract

This systematic literature review examines how code generated by artificial intelligence models impacts software and system security. Through analysis of 19 peer-reviewed studies, the authors found broad consensus that **AI models produce code containing vulnerabilities** despite mitigation attempts. The research identifies specific vulnerability patterns, affected programming languages, and proposes security measures for AI-assisted development workflows.

## 1. Introduction

Organizations increasingly rely on AI tools like GitHub Copilot to enhance software development workflows. However, quality concerns persist. Research by Pearce et al. (2022) determined that "40% of the code suggested by Copilot had vulnerabilities." This creates a paradox: users trust AI-generated code despite security risks.

The central research question guiding this work: **"How does the code generation from AI models impact the cybersecurity of the software process?"**

Key findings indicate:
- AI models consistently produce unsafe code
- Certain vulnerabilities appear more frequently than others
- Some domains face greater risk than others
- Mitigation efforts remain inadequate

## 2. Background

### 2.1 AI Models and Large Language Models

Modern code generation relies on transformer-based architectures. These models:
- Convert text to tokens and map to latent space (encoding)
- Reverse the process during decoding
- Learn from human-generated training data

Pre-training amortizes computational costs through autocompletion tasks. Fine-tuning adapts pre-trained models for specific applications. The temperature hyperparameter controls output predictability — higher values increase randomness.

### 2.2 Code Generation with AI Models

Generative AI tools like Codex, CodeParrot, and StarCoder leverage narrower programming language contexts than general language models. However, this creates risks:

- **Verbatim copying**: Sensitive or dangerous code transfers directly from training data
- **Adaptation errors**: "Minor adaptations" introducing mistakes when transferring code between projects
- **Bad practices propagation**: Well-documented bugs and malpractices from training corpora appear in outputs

Educational concerns include potential for bad security habits among novices, though barriers to entry may lower. Criminal applications remain possible, though defensive cybersecurity applications also exist.

## 3. Research Methodology

### 3.1 Search Strategy and PICO Framework

The study employed PICO (Population, Intervention, Comparison, Outcome) methodology:

- **Population**: Cybersecurity
- **Intervention**: AI models
- **Comparison**: Security issues in AI-generated versus conventional code
- **Outcomes**: Systematic list of security issues and mitigation strategies

Three keyword sets guided the search:
1. Security-related terms (secure, vulnerab*, threat*, exploit, fault*, failure)
2. AI elements (AI, LLM, "large language models")
3. Code generation concepts (creation, writing, production, quality)

Databases searched: IEEE Xplore, ACM, Scopus (including Springer and ScienceDirect).

### 3.2 Inclusion/Exclusion Criteria

**Included:**
- Studies explicitly addressing AI code generation with security focus
- Peer-reviewed publications in English

**Excluded:**
- Non-peer-reviewed work and gray literature
- General AI research without code generation focus
- Secondary research (systematic reviews, summaries)
- Opinion and philosophical papers

### 3.3 Snowballing and Quality Assessment

Forward snowballing identified citing papers using Google Scholar. Quality evaluation applied eight criteria:
- Research method descriptions
- Data collection and analysis procedures
- Security analysis definition
- Threat to validity discussion
- Clear research aims
- Explicit findings statements
- Evidence usability
- Explicit cybersecurity focus

Minimum quality threshold: 50% of possible points (4.0 of 8.0).

### 3.4 Data Extraction

Extracted information included:
- AI model specifications
- Security concern identification methods
- Vulnerability types and classifications
- Known versus novel vulnerabilities
- Programming languages
- Mitigation strategies and their scope

## 4. Results

### 4.1 Sample Selection

Initial search yielded 95 articles across three databases:
- IEEE Xplore: 49
- ACM: 5
- Scopus: 41

Process:
- After deduplication and criteria application: 21 articles
- After forward snowballing: 35 articles
- After full-text review: 23 articles
- After quality assessment: **19 final articles**

Publication timeline: First article in 2022, increasing annually thereafter.

## 5. Discussion

### 5.1 AI Models and Investigation Methods

**Model Coverage:**
- 14 papers (73%) studied OpenAI models, with Codex most popular
- Salesforce's CodeGen and CodeT5 appeared frequently
- Other studied models: Polycoder, GPT-2, StarCoder

**Comparison Challenges:**
Different papers examined different vulnerabilities using different methodologies, making direct comparison difficult. However, notable patterns emerged:

Pearce et al. (2022, 2023) established a baseline: **39% of Python code and 50% of C code** contained Top-25 MITRE vulnerabilities. These findings proved consistent across multiple independent studies.

### 5.2 Programming Language Performance

#### Python
Python emerged as the most studied language (12 papers). Key findings:

- Higher-level language results in fewer vulnerabilities than C
- More training data available improves model performance
- 38.35% vulnerability rate documented (Pearce et al., 2022)
- CWE-089 (SQL injection) particularly problematic: 75% vulnerable in one scenario
- Prompting significantly influences security outcomes

**Critical insight:** Developers using AI assistants for Python generated **more insecure code** than control groups (Perry et al., 2023). 21% of experimental group code was secure versus 43% of controls.

#### C and C++
C appeared in 10 papers (52%). Characteristics:

- Low-level memory management creates larger attack surface
- 50.29% vulnerability rate (Pearce et al., 2022)
- CWE-787 (buffer overflow) most common
- Only 2.2% of code successfully auto-repaired for CWE-787
- Human vs. AI comparison inconclusive: some studies found AI safer

**Key vulnerability:** CWE-787, ranked #1 in MITRE Top-25, appeared across nearly all C-focused studies.

#### Java
Four papers studied Java with diverging focuses:

- **Cryptographic APIs:** Significant accuracy problems (Tony et al., 2022)
- **Simple bugs:** AI models generated twice as many bugs as correct code (Jesse et al., 2023)
- **Vulnerability repair:** Only 20.4% of vulnerabilities fixed by Codex (Wu et al., 2023)
- **Adversarial attacks:** 60% syntactic correctness achieved through manipulation (Jha and Reddy, 2023)

#### Verilog (Hardware Description)
Three papers examined hardware code generation:

- Less training data available than software languages
- Models struggled with functional correctness
- 28.28% vulnerability rate (Pearce et al., 2022)
- CWE-1271 and CWE-1234 (hardware-specific) most studied
- Required different temperature parameters than software languages

**Finding:** Hardware design appears more vulnerable than software generation.

### 5.3 Specific Vulnerabilities and Exploits

**Most Common CWEs:**
- CWE-787 (Out-of-bounds write)
- CWE-089 (SQL injection)
- CWE-079 (Cross-site scripting)
- CWE-125 (Out-of-bounds read)
- CWE-416 (Use-after-free)

**Novel Exploits Identified:**

1. **Membership Inference Attacks (Niu et al., 2023):**
   - Hand-crafted prompts extracted personal data from training sets
   - Results: 2.82% outputs with identifiable information (addresses, emails)
   - 0.78% private information leaks (medical records)
   - 0.64% secret information (keys, passwords)

2. **Adversarial Code Generation (Jha and Reddy, 2023):**
   - "CodeAttack" manipulates models via specially crafted inputs
   - Generated syntactically correct but functionally malicious code
   - Worked across multiple languages and models

3. **Malware Generation:**
   - Models cannot create complete malware from scratch (Botacin, 2023)
   - Can generate snippets assembled into functional malware
   - Basic obfuscation bypasses safeguards (Pa Pa et al., 2023)
   - Up to 400 lines of code achievable with jailbreaking

### 5.4 Mitigation Strategies

#### Dataset Quality
- Better curated training data improves security outcomes
- Trade-off: Smaller high-quality datasets reduce generalization
- Adding diverse language examples helps

#### Training Procedures
- Stricter evaluation of syntactic and semantic correctness
- Fine-tuning with curated security data (He and Vechev, 2023) achieved:
  - CWE-787: 59.1% to 92.3% security rate
  - Sustained improvements across multiple CWEs

#### Generation Improvements
- **Prompting strategies:** Context matters significantly
  - Same scenario with different prompts: 0% to 94% vulnerability rates
  - Documentation and comments improve outcomes
- **Post-processing:** SVEN approach independently verified outputs
- **Safeguarding:** Limits certain prompts but jailbreaking possible

#### Integration Processes
- Mandatory code review and security verification
- Static analysis and automated testing
- Maintain skepticism toward AI-generated code
- Adjusted review methodologies for AI-specific vulnerability patterns

#### User Education
- Limited research on effectiveness
- Skilled developers better positioned to identify and fix issues
- Security vulnerabilities harder to spot than functional bugs
- Combination approach (AI + verification tools) recommended

## 6. Threats to Validity

**Sample Representativeness:**
- Daily publication increases make snapshot samples time-dependent
- Open-source platforms (ArXiv) required manual peer-review verification
- Different "code generation" definitions produce varying samples

**Researcher Bias:**
- Expertise influenced classification and extraction
- Mitigation: Used established taxonomies (Wieringa et al., 2006; MITRE classifications)

**Methodology Heterogeneity:**
- Papers used different vulnerability selection approaches
- Standardized comparison approaches needed for future work
- Open-source vs. proprietary model comparisons limited

## 7. Conclusions

This systematic review confirms that **AI code generation tools produce vulnerable code** across programming languages and models. Key takeaways:

1. **Vulnerability is widespread:** 38-50% of generated code contains Top-25 MITRE vulnerabilities depending on language

2. **Language matters significantly:** Python safer than C; hardware design most problematic

3. **No mitigation is complete:** Technical safeguards help but do not eliminate risk

4. **Human factors critical:** Developer education, trust calibration, and code review processes essential

5. **Model selection important:** OpenAI's Codex generally outperforms alternatives, but all models produce vulnerable code

6. **Future research needed:**
   - Standardized evaluation frameworks
   - Comprehensive model comparisons
   - Jailbreak and exploit prevention
   - Long-term developer behavior studies

**Final statement:** Organizations deploying AI code generation must implement dedicated security measures and maintain healthy skepticism toward generated code. Current tools cannot be trusted to produce secure code without additional verification.
